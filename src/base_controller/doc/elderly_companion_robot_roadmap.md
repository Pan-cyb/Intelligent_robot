# 养老陪伴型机器人项目总纲

本文档作为后续功能实现路线的总纲，目标是把当前移动机器人底盘、激光雷达、建图导航能力，扩展为面向居家养老场景的陪伴与安全守护机器人。

当前仓库已有基础：

- `base_controller`：底盘串口通信、`cmd_vel` 下发、`odom` 发布、`odom -> base_link` TF。
- `ldlidar_ros2`：激光雷达驱动，发布激光扫描数据。
- `mapping.launch.py`：激光建图流程。
- `navigation.launch.py`：Nav2 地图导航流程。

后续需要新增能力：

- 定时任务和生活作息提醒。
- 室内全图导航和定点到达。
- 老人识别、跟随和丢失重识别。
- 摔倒、长时间静止、异常姿态等风险检测。
- 药箱给药或药物提醒执行机构。
- 电话、短信、网络报警。
- 语音交互和大模型陪伴对话。
- 系统级任务调度、安全状态机和日志记录。

## 1. 功能阐述

### 1.1 基础移动能力

机器人需要具备稳定的移动底盘能力：

- 接收上层速度控制指令 `/cmd_vel`。
- 通过串口发送速度到底层 STM32 或其他电机控制器。
- 从底层读取里程计数据。
- 发布 `/odom`。
- 发布 `odom -> base_link` TF。
- 配合激光雷达完成建图、定位和导航。

这一部分由当前 `base_controller` 和 `ldlidar_ros2` 提供基础。

### 1.2 全图导航能力

机器人需要能在家庭地图中到达指定地点，例如：

- 卧室。
- 客厅。
- 药箱位置。
- 充电桩。
- 老人常待区域。
- 紧急呼叫点或门口。

全图导航用于“明确目标点”的任务，例如早上从充电桩出发到卧室叫醒老人，或者接到风险事件后移动到老人附近。

### 1.3 老人跟随能力

机器人需要在老人活动时保持跟随：

- 识别目标老人。
- 估计老人相对机器人方位和距离。
- 控制机器人保持安全跟随距离。
- 避免贴近、碰撞和挡路。
- 老人被遮挡或离开视野时进入搜索或等待状态。

跟随能力用于“目标是人而不是地图坐标”的任务。此时不能只依赖 Nav2 全图目标点，而要使用视觉或多传感器感知实时更新目标位置。

### 1.4 安全风险检测

机器人需要持续关注老人状态，重点检测：

- 摔倒。
- 坐地不起。
- 长时间静止。
- 异常弯腰或趴伏。
- 离开安全区域。
- 夜间异常活动。

第一阶段建议优先实现摔倒检测和长时间静止检测。摔倒检测可基于视觉人体姿态、人体框长宽比、关键点高度变化和地面接触状态综合判断。

### 1.5 药箱给药和提醒

机器人需要支持用药相关功能：

- 定时提醒吃药。
- 导航到老人附近。
- 语音确认老人是否需要药。
- 控制药箱执行机构打开指定药格或推出药物。
- 记录给药时间和老人确认状态。

这里要区分“提醒吃药”和“自动给药”。自动给药属于高风险功能，必须加入确认逻辑、药格状态检测和异常保护。

### 1.6 报警和通知

发生危险事件时，机器人需要按风险等级处理：

- 本地语音询问老人是否需要帮助。
- 如果老人确认需要帮助，立即报警或通知家属。
- 如果老人无响应，进入倒计时，倒计时结束后自动报警。
- 发送事件类型、时间、位置和图片或短视频证据。

报警通道可以分阶段实现：

- 第一阶段：本地日志和屏幕/终端提示。
- 第二阶段：HTTP 接口通知家属手机应用或服务器。
- 第三阶段：短信、电话或第三方平台报警。

### 1.7 AI 大模型陪伴对话

机器人需要能和老人自然交流：

- 语音唤醒。
- 语音识别。
- 对话管理。
- 调用大模型生成回复。
- 语音合成播放。
- 结合机器人状态回答，例如今天是否吃药、当前时间、天气、日程等。

陪伴对话不能直接绕过安全策略。大模型只负责语言理解和回复生成，涉及移动、给药、报警等动作时，应由任务调度器做权限判断和二次确认。

## 2. 工作流程和线程模型

### 2.1 顶层任务状态机

建议用一个 `task_manager` 作为最高层调度节点，维护机器人当前工作模式：

```text
IDLE 空闲
SCHEDULED_TASK 定时任务
NAVIGATION 全图导航
FOLLOWING 老人跟随
MONITORING 安全监护
MEDICINE 用药任务
EMERGENCY 紧急处理
CONVERSATION 对话陪伴
CHARGING 充电
MANUAL 手动控制
FAULT 故障
```

状态优先级建议：

```text
EMERGENCY > MANUAL > MEDICINE > FOLLOWING > NAVIGATION > CONVERSATION > SCHEDULED_TASK > CHARGING > IDLE
```

紧急事件永远优先。手动控制用于调试和接管。对话陪伴不应阻塞安全监护。

### 2.2 什么时候使用全图导航

全图导航适合目标是固定地图点或全局坐标的场景。

典型使用场景：

- 从充电桩去卧室叫醒老人。
- 去药箱位置取药或靠近药箱执行机构。
- 去老人最后一次检测到的位置。
- 去客厅、厨房、卧室等预设房间。
- 回充电桩。
- 巡检固定路线。

全图导航依赖：

- 静态地图。
- AMCL 或 SLAM 定位。
- 激光雷达避障。
- Nav2 planner/controller。
- `base_controller` 发布稳定的 `/odom` 和 TF。

全图导航的输出通常是 `/cmd_vel`，由 `base_controller` 下发到底盘。

### 2.3 什么时候使用老人跟随

老人跟随适合目标是动态人体的场景。

典型使用场景：

- 老人在家中移动，机器人需要陪同。
- 老人起床后，机器人跟随到客厅。
- 安全监护模式下保持老人处于视觉范围内。
- 老人需要语音交流时，机器人保持合适距离。

跟随模式下，不建议持续向 Nav2 发送远距离全图目标点。更合理的做法是：

- 视觉节点输出老人相对位姿。
- 跟随控制节点计算局部速度。
- 必要时结合局部避障或 Nav2 controller。
- 直接发布跟随速度到 `/cmd_vel`，或通过一个速度仲裁节点输出最终 `/cmd_vel`。

如果老人离机器人较远，或者老人离开当前房间，则可以从跟随切换回全图导航：先导航到老人最后已知地图位置，再恢复视觉跟随。

### 2.4 全图导航和跟随的切换

推荐切换逻辑：

```text
目标是房间/地图点 -> 全图导航
目标是视野中的老人 -> 老人跟随
老人丢失但有最后地图位置 -> 全图导航到最后位置
到达最后位置仍未发现老人 -> 原地搜索
搜索失败 -> 通知家属或进入异常状态
```

典型早晨叫醒流程：

```text
定时器触发 07:00
task_manager 创建叫醒任务
机器人从充电桩全图导航到卧室
到达卧室门口或床边目标点
语音播报叫醒
视觉检测老人是否起身
若老人起身并进入视野，切换到跟随
若老人无响应，重复提醒或通知家属
```

典型陪伴跟随流程：

```text
老人进入活动时间
视觉检测到老人
进入 FOLLOWING
跟随节点持续输出局部速度
安全监护节点并行检测摔倒
老人坐下或停留，机器人保持观察距离
老人丢失，进入搜索或导航到最后位置
```

典型摔倒应急流程：

```text
视觉节点检测到疑似摔倒
risk_detector 连续多帧确认
task_manager 切换 EMERGENCY
停止普通导航和跟随
机器人保持安全距离或导航到可观察位置
语音询问老人是否需要帮助
等待确认或倒计时
无响应或确认需要帮助
触发报警通知
必要时打开药箱或急救物品仓
记录事件日志
```

### 2.5 推荐线程和执行器模型

ROS2 中建议按节点拆分，而不是把所有功能写进一个大节点。

关键原则：

- 底盘控制保持高频稳定，不被视觉和大模型阻塞。
- 视觉推理可以低频但要稳定。
- 大模型调用可能很慢，必须异步执行。
- 紧急状态判断不能依赖大模型返回。
- 所有运动指令必须经过速度仲裁和安全层。

推荐执行方式：

```text
base_controller
  50Hz 定时器
  负责串口收发、odom、TF

lidar_driver
  雷达数据线程
  发布 /scan

nav2
  planner/controller/bt_navigator/lifecycle
  负责全图导航

vision_perception
  摄像头采集线程
  视觉推理线程
  发布人体检测、姿态、摔倒候选

person_following
  10-30Hz 控制循环
  根据老人相对位姿生成跟随速度

risk_detector
  5-15Hz 状态判断
  融合视觉、速度、时间窗口

task_manager
  事件驱动
  管理模式切换和任务优先级

voice_dialogue
  ASR/TTS/LLM 异步线程
  负责对话，不直接控制底盘

safety_supervisor
  20-50Hz
  速度限幅、急停、碰撞保护
```

### 2.6 运动指令仲裁

项目后期不应让多个节点同时直接发布 `/cmd_vel`。建议增加一个速度仲裁节点：

```text
nav2 cmd_vel      -> /cmd_vel_nav
following cmd_vel -> /cmd_vel_follow
manual cmd_vel    -> /cmd_vel_manual
safety stop       -> /safety_stop

velocity_mux 输出 -> /cmd_vel
```

优先级建议：

```text
急停 > 手动 > 应急靠近/避让 > 跟随 > 全图导航 > 巡逻
```

这样可以避免 Nav2、跟随、巡逻、手柄同时抢 `/cmd_vel`。

## 3. 路线实现方案和模块构成

### 3.1 模块总览

建议后续项目按以下 ROS2 包或模块拆分：

```text
base_controller
  底盘串口、odom、TF

ldlidar_ros2
  激光雷达驱动

robot_navigation
  Nav2 启动、地图、定位、导航目标管理

robot_bringup
  总启动文件、参数组合、设备启动顺序

task_manager
  顶层任务状态机、任务优先级、模式切换

vision_perception
  摄像头、人体检测、人体跟踪、姿态估计

risk_detector
  摔倒检测、异常静止检测、风险等级判断

person_following
  老人跟随控制

velocity_mux
  多来源速度仲裁和安全限速

medicine_manager
  用药日程、药箱控制、给药确认

emergency_manager
  报警流程、家属通知、事件记录

voice_dialogue
  唤醒词、语音识别、TTS、大模型对话

memory_profile
  老人档案、日程、用药记录、偏好和历史事件

web_or_app_bridge
  家属端接口、远程配置、事件推送
```

### 3.2 传感器和执行器建议

基础传感器：

- 2D 激光雷达：建图、定位、避障。
- RGB 摄像头：人体检测、摔倒检测、对话场景理解。
- 深度相机或 RGBD 摄像头：更稳定地估计人与机器人距离。
- 麦克风阵列：语音唤醒和语音识别。
- 扬声器：语音播报和陪伴对话。
- IMU：辅助判断机器人自身姿态和运动状态。
- 底盘编码器：里程计。

执行器：

- 移动底盘。
- 药箱舵机、电磁锁或步进电机。
- 急停按钮。
- 指示灯或屏幕。
- 充电对接机构。

底盘与导航：

```text

```

```text
/robot_mode               自定义 RobotMode

```

跟随控制：

```text

```

语音和对话：

```text

```

```text

/alarm_command            自定义 AlarmCommand

```

### 3.4 第一阶段：稳定移动底盘和导航

目标：

- 当前底盘串口协议稳定。
- 激光建图可用。
- Nav2 能从 A 点到 B 点。
- RViz 发布目标点后机器人能到达。

任务：

1. 整理 `base_controller`，把串口名、波特率改成 ROS 参数。
2. 去掉或参数化自动巡航，避免默认抢占导航。
3. 增加串口断线日志和重连策略。
4. 完成家庭测试地图。
5. 标注卧室、客厅、药箱、充电桩等命名点。
6. 验证 `/odom`、`/tf`、`/scan`、Nav2 的稳定性。

验收：

- 能连续运行 30 分钟不崩溃。
- RViz 发送 10 次目标点，成功率达到可接受水平。
- 障碍物附近不会明显撞击。

### 3.5 第二阶段：任务管理和定时叫醒

目标：

- 能按时间触发任务。
- 能导航到卧室。
- 能通过语音或蜂鸣器叫醒老人。

任务：

1. 新建 `task_manager`。
2. 新建命名地点配置，例如 `bedroom`, `living_room`, `charger`。
3. 实现定时任务，例如每天 07:00 叫醒。
4. `task_manager` 调用 Nav2 action 导航到卧室。
5. 到达后调用语音播报节点。
6. 加入任务取消、超时、失败重试。

验收：

- 到点后机器人自动去卧室目标点。
- 到达后播报固定语音。
- 导航失败能重试或上报失败。

### 3.6 第三阶段：视觉老人检测和跟随

目标：

- 摄像头识别老人。
- 机器人能保持合适距离跟随。
- 跟随过程中不会直接撞人。

任务：

1. 新建 `vision_perception`，接入摄像头。
2. 使用人体检测模型输出人体框。
3. 加入目标选择逻辑，锁定老人目标。
4. 如果有深度相机，输出相对距离；如果只有 RGB，需要结合框大小估计距离。
5. 新建 `person_following`，根据目标偏移生成 `/cmd_vel_follow`。
6. 新建 `velocity_mux`，仲裁 Nav2 和跟随速度。
7. 加入目标丢失处理。

验收：

- 老人在 1-3 米范围内移动时，机器人能跟随。
- 老人停止时，机器人保持安全距离。
- 目标短暂遮挡后能恢复。

### 3.7 第四阶段：摔倒和异常行为检测

目标：

- 能检测疑似摔倒。
- 能降低误报。
- 能触发应急流程。

任务：

1. 基于人体姿态估计或人体框变化实现摔倒候选。
2. 用时间窗口确认事件，避免单帧误报。
3. 融合人体高度、框长宽比、关键点角度、地面接触时间。
4. 新建 `risk_detector`，输出风险等级。
5. `task_manager` 接收风险事件后进入 `EMERGENCY`。
6. 播放语音询问：“您是否需要帮助？”
7. 等待老人语音或按钮确认。
8. 无响应则触发报警。

验收：

- 模拟摔倒能触发风险事件。
- 正常坐下、弯腰不应频繁误报。
- 紧急状态能停止跟随和普通导航。

### 3.8 第五阶段：药箱和报警

目标：

- 支持用药提醒。
- 支持高风险事件通知家属。

任务：

1. 新建 `medicine_manager`。
2. 建立用药日程配置。
3. 药箱执行机构接入 STM32 或独立 MCU。
4. 加入药格编号、开关状态、给药确认。
5. 新建 `emergency_manager`。
6. 对接 HTTP、短信、电话或家属端服务。
7. 保存事件日志，包括时间、地点、事件类型和处理结果。

验收：

- 到点提醒吃药。
- 给药动作有确认和失败处理。
- 摔倒无响应后能发送报警通知。

### 3.9 第六阶段：AI 对话陪伴

目标：

- 老人能通过语音和机器人交流。
- 机器人能进行陪伴式回复。
- 对话能结合任务和安全状态。

任务：

1. 新建 `voice_dialogue`。
2. 接入唤醒词。
3. 接入 ASR。
4. 接入 TTS。
5. 接入大模型 API 或本地大模型。
6. 设计工具调用接口，例如查询日程、创建提醒、询问天气、触发导航请求。
7. 大模型输出动作意图时，交给 `task_manager` 二次确认。

验收：

- 能完成基本问答。
- 能提醒日程和用药。
- 不会因为大模型误判直接执行危险动作。

## 4. 关键安全原则

1. 大模型不直接控制底盘、药箱和报警。
2. 所有移动指令必须经过 `velocity_mux` 和安全限速。
3. 紧急停止优先级最高。
4. 给药必须有确认、日志和失败处理。
5. 摔倒报警必须考虑误报和无响应场景。
6. 跟随模式要限制最大速度和最小距离。
7. 所有关键任务要有超时和回退。
8. 任何网络服务不可用时，本地安全功能仍应能运行。

## 5. 推荐近期路线

近期不要同时做所有功能。推荐按下面顺序推进：

1. 稳定底盘、雷达、建图、Nav2。
2. 增加 `task_manager`，实现定时导航到卧室和语音播报。
3. 增加摄像头人体检测。
4. 实现老人跟随。
5. 实现摔倒检测和应急状态机。
6. 实现药箱控制和报警。
7. 最后接入大模型对话。

核心原因是：移动能力和状态机是整个项目的骨架，视觉和大模型都是上层能力。底盘、定位、导航和任务调度不稳定时，越早接入复杂 AI，系统越难调试。
